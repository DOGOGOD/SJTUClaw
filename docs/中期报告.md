# SJTUClaw — Minimal AI Agent Runtime

SJTUClaw 是一个面向教学的渐进式 AI Agent 运行环境。它从最基础的 LLM API 调用出发，逐步实现多轮对话、多 Session 管理、上下文压缩（Compaction）、工具调用（Tool Calling）、Gateway 服务层、QQ 机器人通道、定时任务调度、Workspace 管理、高级工具与审批系统，最终形成一套完整的 Skill 系统。

项目以 Python 实现，采用 FastAPI 构建 Gateway，原生支持 OpenAI 兼容的 function calling，提供 Web UI 和 QQ Bot 两种图形化交互入口。

## 快速开始

### 环境要求

- Python >= 3.11

### 安装

```bash
# 克隆仓库
cd SJTUClaw

# 创建虚拟环境（推荐）
python -m venv .venv
source .venv/bin/activate   # Linux/macOS
# .venv\Scripts\activate    # Windows

# 安装依赖
pip install -r requirements.txt

# 注册 sjtuclaw 命令（pip 可编辑安装）
pip install -e .
```

这样安装后，`sjtuclaw` 命令即可在任意路径下使用。

### 配置

```bash
# 方式一：交互式配置向导（推荐，安装后可一键完成 LLM + QQ Bot 配置）
sjtuclaw setup

# 方式二：手动复制模板并编辑
cp .env.example .env
# 编辑 .env，至少填写以下三项：
# LLM_API_KEY=sk-your-api-key-here
# LLM_BASE_URL=https://api.openai.com/v1
# LLM_MODEL=gpt-4o
```

### 运行

```bash
# ---- sjtuclaw 命令（推荐，pip install -e . 后可用）----
sjtuclaw chat                  # 启动 CLI 交互对话（默认命令）
sjtuclaw gateway               # 启动 HTTP Gateway + Web UI + REST API
sjtuclaw setup                 # 交互式配置向导（LLM + QQ Bot 一键配置）

# ---- 或通过 Python 模块方式启动（无需安装）----
python -m claw.main            # 等同于 sjtuclaw chat
python -m claw.gateway         # 等同于 sjtuclaw gateway
# 访问 http://127.0.0.1:8000
```

## 模块详解

### 1. 配置模块 `claw/config.py`

- 从 `.env` 文件或系统环境变量读取所有配置
- 提供 `LLMConfig`（LLM 连接参数）、`CompactionConfig`（压缩参数）、`HeartbeatConfig`（心跳参数）、`QQChannelConfig`（QQ 机器人配置）等数据类
- 配置缺失时给出清晰的中文错误提示
- 支持 UTF-8/GBK 双编码兼容（中文 Windows 环境）

### 2. LLM 客户端 `claw/llm/`

**`client.py`**：OpenAI 兼容的 Chat Completions API 封装
- `chat(messages)` — 纯文本对话
- `chat_with_tools(messages, tools)` — 原生 function calling 模式
- 自动重试（指数退避，最多 2 次）处理网络瞬断、429、5xx
- 请求超时保护（默认 120s）
- 密钥信息自动脱敏（日志安全）

**`protocol.py`**：模型输出协议解析
- 优先使用 API 原生 `tool_calls`
- 回退到 JSON 协议解析（`{type: "tool_call", tool: ..., args: ...}`）
- 支持 `tool_call`（单工具）、`tool_calls`（多工具）、`final`（最终回答）三种格式
- 容忍模型在 JSON 前后混入解释文本（通过括号平衡算法提取）

### 3. Agent Loop `claw/agent/`

**核心入口 `run_agent_turn(session_id, user_message, ...)`** — 系统中 **唯一** 调用 LLM 的地方，所有入口（CLI、Gateway、Scheduler、QQ Bot）都通过它路由。

核心循环逻辑：
```
user message → append to session → loop:
  buildContext → callLLM →
    if final → save & return
    if tool_call → execute tools → append results → continue
```

关键设计：
- **Approval 门控**：write/shell 级别工具执行前创建审批请求并阻塞等待用户决定
- **并行工具执行**：所有只读且并发安全的工具通过 `ThreadPoolExecutor` 并行执行
- **工具结果去重缓存**：同一轮内相同参数的只读工具调用使用缓存结果
- **拒绝循环保护**：同一操作被连续拒绝 3 次后强制停止
- **迭代预算**：通过 `IterationBudget` 和 `_MAX_AGENT_ITERATIONS` 双保险防无限循环
- **事件回调**：支持 `ThinkingEvent`、`ToolCallStartEvent`、`ToolCallEndEvent`、`FinalEvent`、`ErrorEvent` 五种事件，用于 SSE 流式推送
- **任务取消**：支持通过 `threading.Event` 实现运行中任务的终止
- **结构化的 Completion Summary**：工具调用结束后自动附加完成统计块

辅助模块：
- `budget.py` — 基于 token 预算控制迭代次数
- `events.py` — 事件数据模型
- `health.py` — Loop 健康状态监控与告警
- `metrics.py` — 每轮性能指标（LLM 调用次数、工具调用次数、延迟等）
- `turn_context.py` — 单轮上下文封装

### 4. Session 管理 `claw/session/`

**`models.py`** — `Session` 和 `Message` 数据类
- `Session` 字段：`session_id`, `title`, `messages[]`, `summary`, `last_consolidated`, `metadata`, `skill_usage[]`
- `Message` 字段：`role`, `content`, `tool_calls`, `tool_call_id`, `name`, `timestamp`, `_command`, `media`
- 支持原生 function calling 的 `tool_calls` 持久化
- 内置历史消息回放时的清理逻辑（去除运行时上下文标记、图片面包屑等）

**`store.py`** — JSONL 持久化存储（v5）
- 每条 session 是一个 `.jsonl` 文件，首行为元数据，后续每行一个 Message JSON
- Base64 URL-safe 编码生成防碰撞文件名
- 原子写入（tmp + os.replace + fsync）
- 损坏文件自动隔离（`.corrupted-<ts>` 后缀）
- 旧 `session.json` 格式自动迁移
- Crash recovery：运行时检查点与待处理用户轮次标记
- Session forking：支持在指定 user message 前 fork 新 session
- 文件容量强制截断（默认 2000 条消息上限）

**`title.py`** — 首次对话回合自动生成 session 标题

### 5. 上下文构造 `claw/context/`

**`builder.py`** — `ContextBuilder`，负责将存储结构转换为 LLM 的 `messages` 数组
- 组装顺序：identity → soul → memory block → tool contract → skill index → session summary → conversation history
- 缓存机制：system prefix 按 workspace 缓存，memory block 按版本号缓存
- 支持动态 workspace 路径解析（per-session）
- Bootstrap 文件加载（workspace 根目录下的 `AGENTS.md`、`SOUL.md`、`USER.md`）
- Memory context fencing：用 XML 标签包裹记忆上下文，防止模型混淆
- Skill progressive loading：只注入 name+description，完整指令按需加载

**`compaction.py`** — 长对话压缩
- 基于 token 阈值触发（可配置）
- 多轮 token 预算合并（最多 5 轮）
- 用户轮次边界检测（不在中间断开对话轮次）
- 失败保护：压缩失败时绝不删除原始消息
- Cooldown 机制：压缩失败后 10 分钟内不再尝试
- 空闲 session 自动压缩（可配置 TTL，默认 60 分钟）
- 大型工具输出预剪枝

**`compaction_worker.py`** — 异步压缩工作线程
- 后台线程池执行压缩，不阻塞主流程

**`governance.py`** — 上下文治理
- 修复消息对（去除孤立 tool result，回填缺失结果）
- 控制 in-flight 上下文溢出
- 单次最大 tool result 8000 字符限制

**`token_counter.py`** — 基于 tiktoken 的精确 token 计数

**`budget.py`** — 上下文 token 预算估算

### 6. 长期记忆 `claw/memory/`

**`store.py`** — Markdown 文件存储
- 目录结构：`data/memory/<category>/<slug>.md`
- YAML frontmatter + Markdown body（人类可读、编辑器友好、Git 可追踪）
- 分类：`user_preference`（用户偏好）、`project`（项目信息）、`decision`（决策记录）、`fact`（一般事实）、`general`（其他）
- 关键字 + 标签 + CJK 字符级评分检索
- 调用次数追踪与最近召回时间记录
- 旧 `memory.json` 自动迁移
- Monotonic version counter 供 ContextBuilder 缓存失效

**`reflection.py`** — 每日记忆反思
- 定时扫描 session 历史，自动提取新记忆
- 支持手动触发即时反思

### 7. 工具系统 `claw/tools/`

**`base.py`** — 工具框架
- `Tool` 数据类：name, description, input_schema, handler, safety_level, concurrency_safe, max_result_chars
- `ToolRegistry`：注册、查找、参数校验、执行、结果自动截断
- `ToolResult`：统一的 ok/content/error 结果结构
- `ToolGuardrails`：每轮单工具最多 50 次调用，总计最多 200 次
- `RequestContext` / `ContextAware`：支持工具感知当前 session/channel
- Workspace 边界违规检测与 SSRF 防护分类
- `@register_tool` 装饰器支持自动发现
- `prepare_call` 钩子用于执行前验证

**`readonly.py`** — 只读工具
- `current_time` — 获取当前时间（支持 IANA 时区）
- `list_dir` — 列出目录内容（含文件大小）
- `read_file` — 读取文本文件（64KB 截断保护）

**`update.py`** — 写文件工具
- `create_file` — 创建新文件
- `overwrite_file` — 覆盖文件
- `edit_file` — 编辑文件（依赖 Aider 类库）

**`shell.py`** — Shell 工具（有状态）
- `new_shell` — 启动新 shell（旧 shell 自动退出）
- `run_command` — 在活跃 shell 中执行命令（复用同一个 shell 进程）
- 自动 CWD 边界检查（不能离开 workspace）
- 输出截断保护

**`download.py`** — 下载入口工具
- `create_download` — 为 workspace 文件创建 Gateway 可下载的临时入口

**`attachment.py`** — 附件工具
- `copy_attachment_to_workspace` — 将上传附件拷贝到 workspace

**`cron_tool.py`** — 定时任务创建工具（供 LLM 使用）

**`memory_tools.py`** — 记忆读写工具
- `remember` — 保存长期记忆
- `recall` — 检索长期记忆

**`skills_tool.py`** — 技能浏览工具
- `skills_list` — 列出可用 Skill（name + description），轻量索引
- `skill_view` — 加载 SKILL.md 全文或子文件（references/templates/assets）

**`skill_manager_tool.py`** — 技能管理工具
- `skill_manage` — 6 个操作：create / edit / patch / delete / write_file / remove_file
- 安全级别 write，经过审批系统

### 8. Gateway 网关 `claw/gateway/`

**`server.py`** — FastAPI 应用（v0.3.0）
- 完整的 REST API（/chat, /sessions, /memories, /approvals, /cron, /workspace, /downloads 等）
- SSE 流式端点 `/chat/stream`（实时推送 Agent 思考/工具调用/完成事件）
- QQ Bot 通道集成（通过 lifespan 启动/停止）
- 任务终止端点 `/stop`（支持按 session 或全量取消）
- Web UI 静态文件托管（SPA fallback）
- 自动 session 标题生成（首次对话轮次）
- Cron 服务与 Heartbeat 系统作业
- 每日记忆反思服务

**`middleware.py`** — 网关中间件
- `RateLimitMiddleware` — 滑动窗口限流（60 请求/分钟，10 突发）
- `RequestSizeMiddleware` — 请求体大小检查（默认 10MB）
- `RequestLoggingMiddleware` — 结构化请求日志（含 request ID、耗时）

### 9. QQ Bot 通道 `claw/channels/`

- **`base.py`** — `BaseChannel` 抽象基类（start/stop/send 接口）
- **`qq.py`** — QQ Bot 实现，基于官方 QQ Bot API v2 WebSocket
  - 支持 C2C（私聊）和群聊消息
  - 消息处理器桥接到 Agent Loop
  - Session 映射：QQ chat_id → claw session_id（通过 session metadata 持久化）
  - 支持斜杠命令直接处理（绕过 LLM）
- **`qq_constants.py`** — QQ API 常量（接入点 URL、OP 码等）
- **`qq_crypto.py`** — QQ 安全令牌计算
- **`qq_onboard.py`** — 扫码登录获取 QQ Bot 凭证

### 10. 定时任务调度 `claw/scheduler/`

**`service.py`** — `CronService`（基于 asyncio）
- 支持三种调度模式：`at`（一次性）、`every`（固定间隔）、`cron`（cron 表达式）
- 任务持久化（JSON 文件，原子写入 + fsync）
- 多进程安全（filelock）
- 运行历史记录（最近 20 次，含输出路径）
- 心跳 liveness 监测
- 执行输出持久化（用于审计和依赖链）
- 任务依赖链（depends_on 注入上下文）
- 执行次数限制（repeat_times）
- 崩溃安全：recurring job 执行前预先推进下次触发时间
- 任务暂停/恢复/手动触发

### 11. Skill 系统 `claw/skills/` + `claw/tools/`

**设计理念**：Skill 系统采用 LLM 工具驱动 + 渐进式披露的架构。LLM 通过三个工具自主发现、加载和管理 Skill——`skills_list` 浏览目录，`skill_view` 加载完整指南，`skill_manage` 创建/修改 Skill。

**`registry.py`** — `SkillRegistry`
- 扫描 `skills/` 目录，解析 SKILL.md 的 YAML frontmatter
- 渐进式披露：`list_index()` 仅返回 name+description，完整内容通过 `format_full_content()` 按需加载
- 热重载：`rescan()` 基于 mtime 检测变更并返回 diff
- 依赖检查：`requires: {bins: [...], env: [...]}`

**`usage.py`** — `SkillUsageStore`
- use/view/patch 计数器，持久化到 `.usage.json` 侧文件
- 生命周期状态（active → stale → archived）

### 12. 审批系统 `claw/approval/manager.py`

- `ApprovalManager` — 线程安全的内存审批存储
- `create()` → 注册待审批请求

- 支持设置/查看/取消 workspace

## 模块划分逻辑与核心数据结构

### 模块划分原则

项目遵循 **单一职责 + 依赖倒置** 原则。所有模块通过接口契约交互：

1. **纵向分层**：CLI/Gateway → Agent Loop → LLM Client，上层只能调用下层暴露的公开 API
2. **横向隔离**：Session Store、Memory Store、Tool Registry、Skill Registry 各自独立管理数据
3. **统一入口**：`run_agent_turn()` 是所有用户交互的唯一路由点，禁止绕过它直接调用 LLM

### 核心数据结构

#### Session（会话）
```
Session
├── session_id: str          # 唯一标识
├── title: str               # 显示标题
├── messages: list[Message]  # 消息列表
├── summary: str             # 压缩摘要
├── last_consolidated: int   # 已压缩消息索引
├── metadata: dict           # 运行时元数据
├── skill_usage: list[dict]  # Skill 使用记录
├── created_at: str          # ISO-8601
└── updated_at: str          # ISO-8601
```

#### Message（消息）
```
Message
├── role: str                # "user" | "assistant" | "tool" | "system"
├── content: str             # 消息正文
├── tool_calls: list[dict]?  # 工具调用请求（assistant 角色）
├── tool_call_id: str?       # 工具结果匹配 ID（tool 角色）
├── name: str?               # 工具名（tool 角色）
├── timestamp: str           # 创建时间
├── _command: bool           # 是否为斜杠命令（不发送给 LLM）
└── media: list[str]?        # 图片附件路径
```

#### Tool（工具定义）
```
Tool
├── name: str                # 唯一工具名
├── description: str         # 给模型的说明
├── input_schema: dict       # JSON Schema 参数规范
├── handler: Callable        # 执行函数
├── safety_level: str        # read_only | write | shell | download
├── concurrency_safe: bool   # 是否可并行执行
└── max_result_chars: int    # 结果自动截断长度
```

#### ToolResult（工具结果）
```
ToolResult
├── ok: bool                 # 是否成功
├── content: str?            # 成功时的内容
└── error: str?              # 失败时的错误信息
```

#### MemoryEntry（记忆条目）
```
MemoryEntry
├── memory_id: str           # mem_001 格式
├── content: str             # Markdown 正文
├── category: str            # 分类
├── tags: list[str]          # 标签
├── importance: int          # 1-5 重要度
├── source_session_id: str   # 来源 session
├── created_at / updated_at  # 时间戳
├── last_recalled_at: str    # 最近召回时间
└── recall_count: int        # 召回次数
```

#### SkillInfo（技能信息）
```
SkillInfo
├── name: str                # 技能名称
├── description: str         # 适用场景
├── instructions: str        # 工作流程说明
├── directory: Path          # 技能目录
├── source: str              # 技能来源（"workspace"）
├── always: bool             # 是否自动注入上下文
├── available: bool          # 依赖是否满足
└── missing_deps: str        # 缺失的依赖说明
```

#### CronJob（定时任务）
```
CronJob
├── id: str                  # 唯一标识
├── name: str                # 任务名称
├── schedule: CronSchedule   # {kind, at_ms, every_ms, expr, tz}
├── payload: CronPayload     # {kind, message, session_key, origin_channel, ...}
├── state: CronJobState      # {next_run_at_ms, last_status, run_history[]}
├── enabled: bool
└── delete_after_run: bool
```

#### ApprovalRequest（审批请求）
```
ApprovalRequest
├── approval_id: str         # apr_xxxxx 格式
├── session_id: str
├── tool_name: str           # 请求执行的工具
├── tool_args: dict          # 工具参数
├── status: str              # pending | approved | rejected
└── reject_reason: str       # 拒绝原因
```

### 系统中的作用和使用场景

| 数据结构 | 作用 | 使用场景 |
|---------|------|---------|
| Session + Message | 会话状态持久化 | CLI 对话、Gateway 请求、Scheduler 执行 |
| Tool + ToolResult | 工具执行契约 | 模型请求 → runtime 执行 → 结果反馈 |
| MemoryEntry | 跨 session 长期知识 | 用户偏好记忆、项目背景、决策记录 |
| SkillInfo | 可复用任务能力包 | 课程报告生成、材料汇总、演示大纲 |
| CronJob | 定时任务调度 | 提醒、定期总结、Heartbeat 检查 |
| ApprovalRequest | 安全操作审批 | 文件写入、Shell 命令执行前的用户确认 |
| RequestContext | 工具路由上下文 | 工具感知当前 session/channel 以正确路由 |

| 变量 | 默认值 | 说明 |

### 使用 Skill

Skill 通过 LLM 工具驱动，用户无需记忆命令——直接在对话中描述需求即可：

```text
# 对话中描述需求，LLM 自动查找合适的 Skill：

  → LLM 调用 skills_list() → 看到 course-report 匹配
  → LLM 调用 skill_view("course-report") → 按指南生成报告

  → 经过审批 → 保存为 skills/deploy-flow/SKILL.md
```

## 注意事项

1. **API Key 安全**：`.env` 文件已在 `.gitignore` 中排除，切勿将真实密钥提交到仓库。项目代码中包含自动脱敏逻辑，但请仔细检查日志输出。
2. **数据存储**：运行时数据（sessions、memory、cron jobs）存储在 `data/` 目录下。迁移环境时需要保留此目录。
3. **Workspace 边界**：Write/Shell 工具只能在 workspace 内操作。通过 `/workspace set` 设置当前工作区后生效。
4. **审批超时**：默认审批等待时间为 300 秒。超时后自动视为拒绝，操作不会执行。
5. **压缩失败保护**：上下文压缩（Compaction）失败时，原始消息不会被删除。压缩是安全可逆的操作。
6. **并发限制**：Gateway 自带限流（60 请求/分钟），如需调整可修改 `claw/gateway/middleware.py` 中的常量。
7. **QQ Bot 限制**：官方 QQ Bot API 有频率限制和消息格式限制，详见 QQ 开放平台文档。
8. **模型兼容性**：项目使用 OpenAI 兼容 API，理论上兼容所有支持 `/v1/chat/completions` 的服务（如 vLLM、Ollama、LM Studio 等）。
